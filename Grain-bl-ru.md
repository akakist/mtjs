
# Grain: Операционная система для распределённой торговли

## 1. Высокая скорость обновления данных

- **Гранулы** — минимальные единицы состояния (строка заказа, остаток товара, профиль пользователя).
- Обновление **только изменённых листьев** и их родителей → `O(log N)`.
- Нет перестроения всего дерева состояния.

> Позволяет поддерживать высокую частоту транзакций без глобального пересчёта.

## 2. Скорость выполнения контрактов

- JS-контракты (QuickJS) в изолированном окружении.
- **~20 µs на вызов метода** на одном потоке.
- Хот-контракты — кешируются (~15 µs), холодные — загружаются (~70 µs).

> 1 поток → ~50k вызовов/сек. Горизонтальное масштабирование через многопоточность и шардинг.

## 3. Внутренние вызовы контрактов

- Контракты вызывают друг друга → **экосистема сервисов**:
  - Marketplace вызывает логистический контракт
  - Платежный контракт вызывает стейкинг
  - Аналитика подписывается на события
- Сервисы остаются **слабо связанными** и разделяемыми.

## 4. Шардинг запросов на чтение

- **SELECT** шардируется по нодам.
- Добавление нод → рост пропускной способности на чтение.
- Классическая CQRS‑модель: запись через консенсус, чтение через любой узел.

## 5. Лёгкие ноды на клиенте (мобильные приложения)

- Light node хранит только интересующие гранулы.
- Пример: маркетплейс с 10 млн товаров → полное состояние **~10 ГБ**.
- Телефон с 128–256 ГБ справляется без проблем.

## 6. Offline browsing

- Мобильное приложение работает **без сети** с локальной копией гранул.
- Хешированная верификация.
- При возвращении сети — синхронизация изменений.

## 7. Прямой HTTP‑доступ к гранулам

```
GET /grain/{type}/{id}?hash={expected_hash}
```
- Сервер — обычный HTTP‑сервер (nginx, actix).
- Клиент проверяет хеш и может кешировать локально.
- Позволяет строить CDN для состояния, отдавать данные быстрее RPC, не пересчитывать блоки.

## 8. Emit → аналитика (реляционная БД)

- Контракты **эмитят события** при изменении данных.
- События дампятся в реляционную БД (PostgreSQL / ClickHouse).
- Используется для сложных JOIN‑запросов, отчётов, BI.
- Сама сеть не занимается JOIN‑ами — это зона реляционной БД.

## 9. Формирование реляционной базы

- Берётся **состояние на начало периода** (снапшот).
- Догоняются блоки (как события).
- База полностью воспроизводима и детерминирована.

## 10. Доверие к данным в реляционной БД

- Админ БД не может подделать данные без нарушения консенсуса.
- Состояние верифицируется через хеши гранул.
- Любое расхождение → немедленное обнаружение.
- **Никакого доверия к администратору** — только криптографическая гарантия.

## 🛒 Пример: Маркетплейс на Grain

### Проблема Web 2.0
Централизованный маркетплейс требует:
- Дата-центр (например, 64 МВт как у Ozon) → **до $100 млн/мес** на электроэнергию и железо
- Тысячи разработчиков, аналитиков, менеджеров
- Высокий порог входа для продавца (комиссии, штрафы, реклама)
- Монополию данных о продажах, поведении пользователей, логистике

**Продавец — не участник, а клиент системы.**

### Решение на Grain

1. **Смарт-контракт маркетплейса** предоставляет интерфейс для продавцов.
   - Продавцы регистрируются, их адреса заносятся в ACL контракта.
   - Маркетплейс берёт комиссию за исполнение методов контракта (например, **1%**).

2. **Каждый продавец** может написать свой контракт, который вычисляет:
   - цену товара
   - срок доставки
   - доступность

   Маркетплейс опрашивает эти контракты в реальном времени через внутренние вызовы и выдаёт список товаров с разными ценами и сроками поставки.

3. **Комиссия 1%** распределяется:
   - продавцу
   - маркетплейсу
   - владельцам нод (хостам состояния)

   > Никто не работает бесплатно, но монополия невозможна.

4. **Мобильное приложение** как light node:
   - Можно хостить свои товары прямо с телефона.
   - Офлайн‑покупки и просмотр.
   - 10 млн товаров → ~10 ГБ на телефоне.

   > Продавец может работать **со склада и смартфона**.

### Конкуренция на всех уровнях

| Уровень | Кто конкурирует |
|---------|----------------|
| Продавцы | ценой, сроками, качеством контракта |
| Маркетплейсы | любой может запустить свой контракт‑агрегатор |
| Ноды | хостеры состояния получают комиссию |
| Приложения | любой клиент (iOS, Android, Web) |

> **Нет точек монополии.**

### Экономика против Web 2.0

| Статья расходов Web 2.0 | Решение в Grain |
|------------------------|----------------|
| Дата-центр (64 МВт) | Нет центрального ЦОД |
| Хостинг товаров | Клиентские / лёгкие ноды |
| Логистика API | Контракты продавцов |
| Команда разработки | Open source контракты |
| Комиссия маркетплейса | ~1% (гибко, конкурентно) |
| Барьер входа | Ноль: написал контракт → зарегистрировался |

> **Победитель** — тот, у кого лучше контракт, а не тот, кто купил больше рекламы или серверов.

## 🧠 Резюме

**Grain — это не «очередной блокчейн», а прагматичная бизнес-платформа:**

✅ Высокая скорость и низкая стоимость  
✅ Распределённая экономика, дешевле Web 2.0 ЦОД  
✅ Нулевой порог входа для продавцов и маркетплейсов  
✅ Конкуренция контрактов, а не капиталов  
✅ Инструмент для демонополизации торговли  

**Grain = операционная система для распределённого бизнеса.**
